# Phase 0a: Project Overview

## Architecture

**giles** is a Claude Code plugin (v0.4.0) that orchestrates agile sprints with persona-based development. It is not a standalone application -- it is a set of 5 skills that Claude Code invokes via SKILL.md entry points. The plugin manages the full sprint lifecycle: project bootstrap, sprint execution (kickoff/stories/demo/retro), continuous monitoring, milestone releases, and teardown.

Core design principles:
- **Config-driven**: All project-specific values come from `sprint-config/project.toml`, which is generated by `sprint_init.py` and validated by `validate_config.py`.
- **Symlink-based config**: `sprint-config/` contains symlinks pointing to actual project files. Teardown removes symlinks without touching originals. Exception: `giles.md` (the built-in scrum master persona) is copied, not symlinked.
- **GitHub as source of truth**: Local tracking files sync from GitHub issue/PR state.
- **All scripts are stdlib-only Python 3.10+**: No pip dependencies. Custom TOML parser instead of `tomllib`.
- **Idempotent scripts**: All bootstrap and monitoring scripts are safe to re-run.

The skill lifecycle is: `sprint-setup` -> `sprint-run` (repeats per sprint) -> `sprint-release` (at milestone end) -> `sprint-teardown`. `sprint-monitor` runs alongside `sprint-run` via `/loop`.

## Key Observations

1. **Three scripts are undocumented in CLAUDE.md and CHEATSHEET.md:**
   - `scripts/commit.py` (152 lines) -- conventional commit enforcement. Referenced by sprint-release SKILL.md but absent from the scripts tables.
   - `scripts/verify_line_refs.py` (167 lines) -- verifies line-number references in CLAUDE.md/CHEATSHEET.md. Called by `make lint` but absent from the scripts tables.
   - `skills/sprint-release/scripts/release_gate.py` (645 lines) -- the largest skill script. Contains full release pipeline logic (gate validation, version calculation, release notes, publishing). Referenced by sprint-release SKILL.md but absent from the scripts tables in CLAUDE.md and CHEATSHEET.md.

2. **One reference file is undocumented in CLAUDE.md:**
   - `skills/sprint-setup/references/prerequisites-checklist.md` -- exists on disk and is referenced by sprint-setup SKILL.md, but not listed in the CLAUDE.md "Reference Files" table.

3. **evals.json `skill_name` mismatch:** The evals file uses `"skill_name": "sprint-process"` which does not correspond to any actual skill name in plugin.json. The plugin defines five skills: `sprint-setup`, `sprint-run`, `sprint-monitor`, `sprint-release`, `sprint-teardown`.

4. **Test infrastructure uses unittest, not pytest:** Despite having a `.venv` with pytest installed and a `.pytest_cache/` directory, the Makefile and CI both use `python -m unittest discover`. The pytest artifacts appear to be leftover from earlier development.

5. **Significant test coverage:** 10 test files totaling substantial code. `test_gh_interactions.py` (52KB) and `test_pipeline_scripts.py` (56KB) are the largest. All tests use `fake_github.py` to mock the `gh` CLI.

6. **`sprint-teardown` has no local scripts/ or references/ directory.** It delegates to the shared `scripts/sprint_teardown.py`. All other skills have their own scripts/ and/or references/ subdirectories.

7. **Line-number references self-check:** The project has a built-in mechanism (`verify_line_refs.py`, run during `make lint`) to detect stale line-number references in CLAUDE.md and CHEATSHEET.md. This runs in CI, so stale refs would fail the build.

8. **Plugin version is 0.4.0.** The project appears to be pre-1.0, actively developed.

9. **Historical audit artifacts present:** `bug-hunter-prior-2026-03-14/`, `bug-hunter-prior-pass5/`, `audit/`, and `recon/` directories exist, indicating ongoing audit activity. `ADVERSARIAL-REVIEW.md` (30KB) is also present.

10. **`docs/superpowers/` directory is large:** 12 plan files and 5 spec files, some very large (83KB, 72KB, 68KB). These are historical design documents used during development via the "superpowers" plugin workflow.

11. **Two Python versions in use:** `.python-version` says 3.10, but `__pycache__` contains both `.cpython-310.pyc` and `.cpython-312.pyc` files, indicating the developer runs both locally. CI tests 3.10-3.13.

12. **`.worktrees/` directory exists but is empty.** The `.gitignore` ignores `.claude/worktrees/` and `.worktrees/`. This suggests git worktree usage for parallel development.
