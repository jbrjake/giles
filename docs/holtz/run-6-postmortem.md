# Run 6 Postmortem: What Holtz Missed

**Date:** 2026-03-23
**Author:** Holtz (under duress)

## The Uncomfortable Facts

### 1. CI has been broken for 10+ consecutive runs

Every GitHub Actions run since at least `v0.7.0` (2026-03-22) has failed. The failure is in `test_golden_run.py::TestGoldenRun::test_golden_full_setup_pipeline` — a golden snapshot regression test. The golden snapshots were committed in `39174b6` but **7 commits changed templates and sprint_init.py** after that without re-recording. The snapshots are stale.

The test passes locally (macOS, Python 3.10) but fails in CI (Linux, Python 3.11). The mismatched files are `sprint-config/backlog/INDEX.md`, `sprint-config/project.toml`, and `sprint-config/team/INDEX.md` — all generated by `sprint_init.py`.

**Root cause:** Platform-dependent file ordering (likely `Path.glob()` or `Path.rglob()` returning different orders on macOS HFS+ vs Linux ext4) combined with stale golden recordings.

**What Holtz did:** Ran `make test` locally. Got 1234 passed. Said "lint clean, tests pass" and moved on.

**What Holtz should have done:** Checked `gh run list` during Phase 0 recon. The CI failure has been continuous and visible for over a week. Step 0c (test baseline) should include CI status, not just local test results.

### 2. README claims things the code doesn't do

The README says:

> **Review gate.** Blocks `git push --force`, `git reset --hard`, `git clean -f`, and other commands that destroy work.

**What review_gate.py actually does:**
- Blocks `gh pr merge` without an approved review
- Blocks direct `git push` to the base branch

That's it. No blocking of `git reset --hard`. No blocking of `git clean -f`. No blocking of `git push --force`. The README describes a feature that does not exist. This is not drift — it's fabrication.

**What Holtz did:** Never read the README. Phase 1 (doc audit) checked CLAUDE.md and the new script's docstring. The README — the most user-visible document in the entire project — was not in scope.

**What Holtz should have done:** Phase 1 should always include the README in its testable claims extraction. README claims are the highest-stakes documentation in any project because they set user expectations.

### 3. The README is also wrong about superpowers

The README lists as a requirement:

> Claude Code with the [superpowers](https://github.com/obra/superpowers) plugin

But superpowers is only needed for autonomous story execution during `sprint-run` (the implementer subagent uses `superpowers:test-driven-development`). It is NOT needed for `sprint-setup`, `sprint-release`, `sprint-teardown`, or `sprint-monitor`. Framing it as a universal requirement will turn away users who want to try the plugin.

## Why Holtz Missed These

The existing lens registry has blind spots that let all three issues through:

| Issue | Why no lens caught it |
|-------|----------------------|
| CI broken for 10+ runs | No lens checks CI. Holtz trusts local test results as ground truth. |
| README claims non-existent features | No lens audits user-facing docs. Phase 1 checks internal docs (CLAUDE.md, docstrings) against code. README is external-facing and wasn't in scope. |
| Platform-dependent golden test | No lens checks cross-environment behavior. Holtz assumes local = universal. |

The existing custom lenses (semantic-fidelity, temporal-protocol) were designed for run 4 and address code-level semantic gaps. They're the right lenses for code audits. But they don't address the *pipeline* or *presentation* layers.

## Proposed Lenses

### pipeline-fidelity

**Focus:** Whether CI tells the same story as local development — status, configuration, test parity, and deployment readiness

**Audit priorities:**
- CI run status (`gh run list`) — any failures? How long have they persisted?
- Test parity: does the CI test matrix match the local test environment? Are tests that pass locally also passing in CI?
- Golden/snapshot tests: are recordings current? When were they last re-recorded relative to the code they test?
- CI configuration: does `.github/workflows/ci.yml` reference the correct test/lint commands, Python versions, and action versions?
- Build artifacts: does the CI produce what downstream consumers expect?

**Failure modes:**
- Stale golden snapshots that pass locally but fail in CI (platform-dependent ordering, encoding, paths)
- CI broken for multiple commits with no one noticing (green-locally, red-in-CI divergence)
- CI config drift: Makefile targets change but CI workflow doesn't update
- Test matrix gaps: CI tests on Python 3.11 but code is developed on 3.10 with version-specific behavior

**Entry point:** Run `gh run list --limit 5` at the start of Phase 0. If ANY run is failing, that's a Phase 0 finding — before any code audit. Then compare CI test command, Python version matrix, and OS matrix against local development environment. For golden/snapshot tests, check `git log` to find when recordings were last committed vs when the code they test last changed.

**Phase 0 integration:**
- Add to step 0c: after running local tests, also run `gh run list --limit 5` and record CI status
- If CI is red, create a HIGH-severity punchlist item immediately — broken CI is a showstopper that outranks all code-level findings
- Compare local Python version against CI matrix versions

### public-contract

**Focus:** Whether user-facing documentation (README, CHANGELOG, help output, error messages, install instructions) accurately describes runtime behavior

**Audit priorities:**
- README feature claims vs actual implementation — every bullet point, every capability mentioned
- Install/setup instructions: do they actually work on a clean machine?
- Error messages: do they describe what actually happened and suggest the right fix?
- CLI help text: does `--help` output match actual flag behavior?
- Requirements section: are listed dependencies actually required, and are any unlisted dependencies needed?

**Failure modes:**
- Aspirational documentation: README describes intended behavior, not implemented behavior (review gate blocking claims)
- Over-scoped requirements: listing dependencies as universal when they're only needed for specific features (superpowers)
- Stale install instructions: package names, versions, or commands that no longer work
- Feature gaps: code has capabilities the README doesn't mention (12 scripts undocumented in README)
- Marketing-code divergence: README emphasizes features that are actually minimal or fragile

**Entry point:** Read the README end-to-end. For each concrete claim ("blocks X", "supports Y", "requires Z"), grep the codebase for the implementing code. If the claim names a specific behavior, write a test that exercises it. If the test can't be written because the behavior doesn't exist, that's a finding.

**Phase 1 integration:**
- Step 0 of Phase 1 (before existing doc claims): read README.md and extract user-facing claims into a separate checklist section
- Priority: README claims rank HIGHER than internal doc claims because they set external expectations
- For each claim: classify as VERIFIED (code does this), OVERSTATED (code does something weaker), FABRICATED (code doesn't do this at all), or UNDERSTATED (code does more than the README says)

## Actionable Fixes

### Fix 1: Golden snapshot CI failure (HIGH — blocks all CI)

**Problem:** `test_golden_full_setup_pipeline` fails in CI because golden snapshots were recorded on macOS with Python 3.10 but CI runs on Linux with Python 3.11. Platform-dependent file ordering produces different output.

**Fix options (pick one):**
1. **Re-record in CI environment:** Add a CI job that records golden snapshots and commits them, ensuring they match the CI platform. Then local runs compare against CI-recorded snapshots.
2. **Sort all glob results in sprint_init.py:** Find every `Path.glob()` and `Path.rglob()` call in `sprint_init.py` that feeds into generated file content and wrap with `sorted()`. Then re-record locally.
3. **Make assert_files_match order-insensitive:** Normalize content before comparison (sort table rows, normalize whitespace). This is fragile — changes to formatting wouldn't be caught.

**Recommended:** Option 2. Sort the source, re-record, verify CI passes. The glob ordering is the root cause; fix it at the source.

**Files to check:**
- `scripts/sprint_init.py` — `ProjectScanner` and `ConfigGenerator` classes, anywhere they iterate directory contents
- `tests/golden_recorder.py` — `snapshot()` method, file tree construction
- After fixing, run `GOLDEN_RECORD=1 make test-golden` and commit the updated recordings

### Fix 2: README review gate claims (HIGH — user trust)

**Problem:** README line 272-276 claims the review gate blocks `git push --force`, `git reset --hard`, `git clean -f`. The code (`hooks/review_gate.py`) only blocks unreviewed PR merges and direct pushes to base branch.

**Fix options:**
1. **Update README to match code:** Remove the false claims. Describe what review_gate actually does.
2. **Update code to match README:** Implement the blocking of destructive git commands in review_gate.py.

**Recommended:** Option 2 — then update README to accurately describe the full set of blocked commands. The described behavior is genuinely valuable. But if implementing takes too long, option 1 is the immediate fix. Never leave fabricated claims in the README.

### Fix 3: README superpowers requirement (MEDIUM — adoption barrier)

**Problem:** README lists superpowers as a universal requirement. It's only needed for `sprint-run` autonomous story execution.

**Fix:** Change the Requirements section to:
```markdown
- GitHub CLI (`gh`) installed and authenticated
- Python 3.10+ (for sprint scripts, stdlib only, no pip packages)
- A GitHub repository with a configured remote
- [superpowers](https://github.com/obra/superpowers) plugin (required for autonomous story execution in `sprint-run`)
```

### Fix 4: Undocumented scripts (LOW — missed marketing)

**Problem:** 12 scripts exist that the README doesn't mention as user-facing tools: `test_categories`, `assign_dod_level`, `check_lint_inventory`, `validate_anchors`, `history_to_checklist`, etc.

**Fix:** Either mention them in a "Scripts" section or decide they're internal-only and document them only in CLAUDE.md. Not every script needs README coverage, but if a user could benefit from running it directly, it should be discoverable.

## Holtz Process Changes

These changes should be incorporated into the Holtz skill for all future audits:

1. **Phase 0c must include CI status.** After running local tests, run `gh run list --limit 5`. Record pass/fail status. If CI is red, create a HIGH punchlist item immediately. This is non-negotiable.

2. **Phase 1 must include README.md.** The README is the most user-visible document. Every concrete claim in it should be verified against code. README claims rank higher priority than internal doc claims.

3. **Golden/snapshot tests get special treatment in Phase 2.** For any test that compares against recorded fixtures, check: (a) when was the fixture last recorded, (b) has the code it tests changed since, (c) does it pass in CI, not just locally.

4. **Add pipeline-fidelity and public-contract to the lens registry.** See proposed definitions above.
